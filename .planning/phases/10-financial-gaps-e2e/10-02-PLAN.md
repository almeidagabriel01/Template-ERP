# Plan 10-02: Proposal Approval → Transaction Sync E2E

**Phase:** 10 — Financial Gaps E2E
**Requirements:** FIN-09
**Depends on:** Phase 3 (proposal-status.spec.ts pattern), Phase 4 (TransactionsPage POM)

## Goal

Add a standalone spec file `e2e/financial/proposal-sync.spec.ts` that validates: creating a proposal via API → approving it via API → navigating to `/transactions` and finding the synced transaction by description matching the proposal title.

## Context

Per D-08/D-09/D-10/D-11/D-12 from Phase 10 CONTEXT.md:

- **Simple single-payment proposal** — no installments, no down payment. Approval creates exactly 1 income transaction.
- **`syncApprovedProposalTransactions`** runs synchronously inside `updateProposal` — after the PUT returns 200, the transaction already exists in Firestore. No polling needed.
- **Transaction description = proposal title** — the sync function sets `description = normalizeProposalTransactionTitle(title)`. Verify by matching on proposal title text in the transactions list.
- **Approval requires status to be "sent" first** — the controller validates status transitions. Create as "draft", transition to "sent", then approve.
- **Cleanup**: Delete the proposal via API. Check whether synced transactions are cascade-deleted; if they persist, delete them individually via API.
- **New file**: `e2e/financial/proposal-sync.spec.ts` — standalone, not appended to existing files.

## Tasks

### Task 1: Create `e2e/financial/proposal-sync.spec.ts`

```typescript
/**
 * FIN-09: Proposal approval → transaction sync E2E test.
 *
 * Validates that approving a proposal triggers syncApprovedProposalTransactions
 * and the resulting income transaction appears in the financial module.
 *
 * Flow (per D-08/D-09/D-10/D-11/D-12):
 *  1. Create a simple single-payment proposal via API (status: "draft").
 *  2. Transition draft → sent via API (required before approval).
 *  3. Transition sent → approved via API — triggers syncApprovedProposalTransactions synchronously.
 *  4. Navigate to /transactions and verify the synced transaction is visible by title match.
 *  5. Cleanup: delete proposal via API (cascade-deletes synced transactions if controller handles it;
 *     otherwise also delete the synced transaction).
 */

import { test, expect } from "../fixtures/auth.fixture";
import { TransactionsPage } from "../pages/transactions.page";
import { signInWithEmailPassword } from "../helpers/firebase-auth-api";
import { USER_ADMIN_ALPHA } from "../seed/data/users";
import { PROPOSAL_ALPHA_DRAFT } from "../seed/data/proposals";

test.describe("FIN-09: Proposal approval → transaction sync", () => {
  test("approving a proposal creates a matching income transaction in the financial module", async ({ authenticatedPage }) => {
    const { idToken } = await signInWithEmailPassword(
      USER_ADMIN_ALPHA.email,
      USER_ADMIN_ALPHA.password,
    );

    const timestamp = Date.now();
    const proposalTitle = `FIN-09 Sync Test ${timestamp}`;

    // Step 1: Create a fresh draft proposal via API.
    // Use PROPOSAL_ALPHA_DRAFT fields for client/items — simple single-item, no installments.
    const createResponse = await authenticatedPage.request.post("/api/backend/v1/proposals", {
      headers: { Authorization: `Bearer ${idToken}` },
      data: {
        title: proposalTitle,
        clientId: PROPOSAL_ALPHA_DRAFT.contactId,
        clientName: PROPOSAL_ALPHA_DRAFT.contactName,
        status: "draft",
        totalValue: PROPOSAL_ALPHA_DRAFT.total,
        products: PROPOSAL_ALPHA_DRAFT.items.map((item) => ({
          id: item.productId,
          name: item.productName,
          quantity: item.quantity,
          unitPrice: item.unitPrice,
          total: item.total,
        })),
      },
    });
    expect(createResponse.status()).toBe(201);
    const createData = await createResponse.json();
    const proposalId = createData.proposalId;
    expect(proposalId).toBeTruthy();

    // Step 2: Transition draft → sent (required before approval — controller validates transition).
    const sentResponse = await authenticatedPage.request.put(
      `/api/backend/v1/proposals/${proposalId}`,
      {
        headers: { Authorization: `Bearer ${idToken}` },
        data: { status: "sent" },
      },
    );
    expect(sentResponse.status()).toBe(200);

    // Step 3: Transition sent → approved — triggers syncApprovedProposalTransactions.
    // The sync runs synchronously before the API returns 200, so no polling needed (D-09 note).
    const approveResponse = await authenticatedPage.request.put(
      `/api/backend/v1/proposals/${proposalId}`,
      {
        headers: { Authorization: `Bearer ${idToken}` },
        data: { status: "approved" },
      },
    );
    expect(approveResponse.status()).toBe(200);

    // Step 4: Navigate to /transactions and find the synced transaction.
    // The sync function sets description = normalizeProposalTransactionTitle(title).
    // Verified in proposals.controller.ts — description matches the proposal title.
    const transactionsPage = new TransactionsPage(authenticatedPage);
    await transactionsPage.goto();
    await transactionsPage.isLoaded();

    // The transaction may appear on the first load since sync is synchronous.
    // If not immediately visible, reload once to account for any cache delay.
    let syncedItem = await transactionsPage.getTransactionByDescription(proposalTitle);
    const isVisible = await syncedItem.isVisible().catch(() => false);
    if (!isVisible) {
      await authenticatedPage.reload();
      await transactionsPage.isLoaded();
      syncedItem = await transactionsPage.getTransactionByDescription(proposalTitle);
    }

    await expect(syncedItem).toBeVisible({ timeout: 10000 });

    // Step 5: Cleanup — delete the proposal via API.
    // If the proposal controller cascades deletion to synced transactions, this is sufficient.
    // If not, we also need to find and delete the synced transaction.
    const deleteProposalResponse = await authenticatedPage.request.delete(
      `/api/backend/v1/proposals/${proposalId}`,
      { headers: { Authorization: `Bearer ${idToken}` } },
    );
    expect([200, 204]).toContain(deleteProposalResponse.status());

    // Verify synced transaction is also gone (cascade check).
    // Reload transactions list — if the transaction persists, delete it via its ID.
    await transactionsPage.goto();
    await transactionsPage.isLoaded();
    const afterDelete = await transactionsPage.getTransactionByDescription(proposalTitle);
    const stillVisible = await afterDelete.isVisible().catch(() => false);

    if (stillVisible) {
      // Cascade did not delete it — clean up via UI delete
      await transactionsPage.deleteTransaction(proposalTitle);
    }
  });
});
```

## Verification Criteria

- FIN-09: After proposal approved via API, `/transactions` shows a transaction whose description matches the proposal title → passes
- Synced transaction amount corresponds to `PROPOSAL_ALPHA_DRAFT.total` (3100.00) — visible in the row
- Cleanup leaves no orphaned test data

## Implementation Notes

- **Status transition sequence**: `draft → sent → approved`. The controller rejects direct `draft → approved` transitions. Two PUT calls are required.
- **Sync is synchronous**: `syncApprovedProposalTransactions` is called inside `updateProposal` before the response is returned. The transaction exists immediately after the PUT returns 200. The reload fallback in the test is a safety net for any stale frontend cache.
- **Description matching**: `normalizeProposalTransactionTitle(title)` may or may not lowercase/trim the title. The test uses the exact proposal title string in `getTransactionByDescription()` which does a substring filter (`hasText`) — this is case-sensitive. If the title is normalized differently, check `proposals.controller.ts:647` and adjust the search string.
- **Cascade deletion**: Whether deleting a proposal also deletes its synced transactions is an open question (D-11 in CONTEXT.md). The test handles both cases: if the transaction is gone after proposal deletion, nothing extra is needed; if it persists, `deleteTransaction()` cleans it up via the UI.
- **New file location**: `e2e/financial/proposal-sync.spec.ts` — consistent with the existing financial spec directory layout.

## Risk

Low-medium — depends on `normalizeProposalTransactionTitle()` behavior and cascade deletion behavior. Both are handled with defensive fallbacks in the test code. Read `proposals.controller.ts` lines 647 and 782 before implementing to confirm the exact description format.
