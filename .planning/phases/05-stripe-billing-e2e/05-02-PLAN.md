**Strategy for avoiding 30s plan cache:** Create a throwaway tenant with a unique ID per test run (e.g., `tenant-billing-${Date.now()}`). This tenant ID was never cached, so the first policy read fetches fresh from Firestore. The throwaway tenant still uses `USER_ADMIN_BETA`'s idToken — the proposals controller reads plan limits from the `tenants/` doc keyed by the token's `tenantId` claim. BUT since `USER_ADMIN_BETA` has `tenantId: "tenant-beta"` baked into their custom claims, we CANNOT use a different tenantId in the API call — the controller will use `req.user.tenantId` which is `"tenant-beta"`.

**Revised approach:** Since the test is bound to `tenant-beta` via the user's claims, use `tenant-beta` directly. To handle the 30s cache, run this test file in isolation (not parallel with subscription.spec.ts tests that also mutate tenant-beta). The `test.describe.configure({ mode: 'serial' })` in this file ensures sequential execution. Add a 31-second wait between plan state changes ONLY if needed (try without first).

**Describe block: "BILL-04: Free plan blocks proposal creation at limit"**
- `test.beforeEach`: Call `seedBillingState(db, "tenant-beta", "free", 5)` to set tenant-beta to free plan with 5 proposals created (at the free limit of 5).
- `test.afterEach`: Call `restoreTenantState(db, "tenant-beta")` to clean up.
- Test 1: "free plan tenant at proposal limit receives 402 with PLAN_LIMIT_PROPOSALS_MONTHLY":
  1. Get idToken via `signInWithEmailPassword(USER_ADMIN_BETA.email, USER_ADMIN_BETA.password)`
  2. POST to `${FUNCTIONS_BASE}/v1/proposals` with `Authorization: Bearer ${idToken}` and minimal valid proposal payload
  3. Assert response status is **402**
  4. Assert response body contains:
     - `code` equal to `"PLAN_LIMIT_PROPOSALS_MONTHLY"`
     - `used` equal to `5`
     - `limit` equal to `5`
     - `tier` equal to `"free"`
  5. Assert response body `message` contains `"Limite de propostas"`

- Test 2: "free plan tenant BELOW limit can still create a proposal":
  1. Call `seedBillingState(db, "tenant-beta", "free", 4)` to set usage to 4 (one below limit)
  2. Get idToken
  3. POST proposal — assert response status is **201** (allowed, one slot remaining)
  4. Clean up created proposal via DELETE `${FUNCTIONS_BASE}/v1/proposals/{id}`

This second test proves the enforcement is not a blanket block but responds to actual usage count, adding confidence that the limit check is correct.

**Describe block: "BILL-05: WhatsApp overage cron"**

`test.beforeAll`:
- Get `db` via `getTestDb()`

- NOTE: Intentionally do NOT set `stripeCustomerId` — controller will push to `errors[]` with `"Missing tenant.stripeCustomerId"`, skipping the real Stripe API call. This is the recommended approach from the RESEARCH.md.

`test.afterAll`:
- Clean up tenant WhatsApp fields: `db.collection("tenants").doc("tenant-beta").update({ whatsappEnabled: admin.firestore.FieldValue.delete(), whatsappAllowOverage: admin.firestore.FieldValue.delete() })`
- Delete the test whatsappUsage doc: `db.collection("whatsappUsage").doc("tenant-beta").collection("months").doc(TEST_MONTH).delete()`

Test 1: "cron processes tenant with overageMessages and reports result" (per D-09):
  1. Seed Firestore with overage usage doc:
     ```typescript
     await db.collection("whatsappUsage").doc("tenant-beta")
       .collection("months").doc(TEST_MONTH).set({
         overageMessages: 50,
         stripeReported: false,
       });
     ```
     NOTE: Collection name is `whatsappUsage` (camelCase, per Research Finding 4 — NOT `whatsapp_usage`).
  2. POST to `${FUNCTIONS_BASE}/internal/cron/whatsapp-overage-report` with headers:
     ```typescript
     {
       "x-cron-secret": CRON_SECRET,
       "Content-Type": "application/json",
     }
     ```
     And body: `{ month: TEST_MONTH }` (per D-09)
  3. Assert response status is 200
  4. Assert response body contains `processed` field (number >= 0) — the cron ran and returned results
  5. Read back the Firestore doc:
     ```typescript
     const usageSnap = await db.collection("whatsappUsage").doc("tenant-beta")
       .collection("months").doc(TEST_MONTH).get();
     ```
  6. Since Stripe call fails (no stripeCustomerId), `stripeReported` should remain `false`. Assert `usageSnap.data()?.stripeReported` is `false`.
  7. Assert response body `errors` array contains an entry with `tenantId: "tenant-beta"` (confirming the cron found and processed this tenant but Stripe call was skipped due to missing stripeCustomerId).

Test 2: "cron skips tenant with stripeReported=true (idempotency)":
  1. Seed Firestore with already-reported usage doc:
     ```typescript
     await db.collection("whatsappUsage").doc("tenant-beta")
       .collection("months").doc(TEST_MONTH).set({
         overageMessages: 50,
         stripeReported: true, // already reported
       });
     ```
  2. POST to the cron endpoint (same as above)
  3. Assert response status is 200
  4. Assert response body `skipped` field is >= 1 (tenant was skipped because already reported)

CRON_SECRET handling (per D-10): The test reads from `process.env.CRON_SECRET` first. If not available, falls back to `"test-cron-secret"`. The executor must verify this matches the Functions emulator env. If neither approach works, read `internal.controller.ts` to see how the secret is validated and adapt.
