Per D-04: Test verifies HTTP response only — `GET /api/backend/proposals/:id/pdf` must return a non-401/403 status with proper content-type. No PDF content inspection.

```typescript
import { test, expect } from "../fixtures/auth.fixture";
import { PROPOSAL_ALPHA_APPROVED } from "../seed/data/proposals";

test.describe("PROP-04: PDF generation endpoint", () => {
  test("returns non-auth-error response for authenticated request", async ({ authenticatedPage }) => {
    const response = await authenticatedPage.request.get(
      `/api/backend/proposals/${PROPOSAL_ALPHA_APPROVED.id}/pdf`
    );

    // In emulator, PDF generation may fail (500) because Playwright/Chromium is not available server-side.
    // Per D-04: we verify the endpoint is reachable and auth works (not 401/403).
    // A 200 means PDF was generated; a 500 means auth passed but generation failed in emulator.
    expect(response.status()).not.toBe(401);
    expect(response.status()).not.toBe(403);

    // If 200, also verify content-type
    if (response.status() === 200) {
      const contentType = response.headers()["content-type"];
      expect(contentType).toContain("application/pdf");
    }
  });
});
```

**File 2: `e2e/proposals/proposal-share.spec.ts`** (PROP-05)

Per D-08: Create share token via authenticated API call, then open `/share/:token` in a fresh unauthenticated browser context.
Per D-09: Assert (a) no redirect to `/login` — URL must remain `/share/:token`, and (b) proposal title/content is visible.

```typescript
import { test, expect } from "../fixtures/auth.fixture";
import { PROPOSAL_ALPHA_APPROVED } from "../seed/data/proposals";

test.describe("PROP-05: Public share link", () => {
  test("share link is accessible without authentication and renders proposal content", async ({
    authenticatedPage,
    browser,
  }) => {
    // Step 1: Create share token via authenticated API (per D-08)
    const shareResponse = await authenticatedPage.request.post(
      `/api/backend/proposals/${PROPOSAL_ALPHA_APPROVED.id}/share-link`
    );
    expect(shareResponse.status()).toBe(200);

    const shareData = await shareResponse.json();
    // Response shape: { shareUrl, token, expiresAt }
    expect(shareData.token).toBeTruthy();
    const token = shareData.token;

    // Step 2: Open share URL in fresh unauthenticated browser context (per D-08)
    // Use browser.newPage() — no storageState, no auth cookies
    const publicPage = await browser.newPage();

    await publicPage.goto(`/share/${token}`);
    await publicPage.waitForLoadState("networkidle");

    // Step 3: Assert no redirect to /login (per D-09)
    const currentUrl = publicPage.url();
    expect(currentUrl).toContain(`/share/${token}`);
    expect(currentUrl).not.toContain("/login");

    // Step 4: Assert proposal content is visible (per D-09)
    // The approved proposal title should be rendered on the public page
    const pageContent = await publicPage.textContent("body");
    expect(pageContent).toContain(PROPOSAL_ALPHA_APPROVED.title);

    await publicPage.close();
  });
});
```

**Key implementation notes:**
- `page.request` inherits the authenticated session cookies from `authenticatedPage` fixture — no manual auth headers needed.
- `browser.newPage()` creates a context without any `storageState` — simulates unauthenticated user (per Pitfall 3 from RESEARCH.md).
- Use `PROPOSAL_ALPHA_APPROVED` for both tests since it's a stable seed proposal that won't be mutated (share link creation is non-destructive, PDF read is non-destructive).
- The share page at `/share/[token]` fetches data via `GET /v1/share/:token` (public endpoint) and renders `ProposalPdfViewer`.

Per D-05: Status changes are driven via backend API (`PUT /api/backend/proposals/:id` with `{ status: "sent" }`), not through kanban UI.
Per D-07: Use seed proposals as starting states where possible. However, to avoid cross-test pollution (Pitfall 1 from RESEARCH.md), create fresh proposals via API for each transition test.

**Strategy:** For each transition test:
1. Create a fresh proposal via `POST /api/backend/proposals` with the starting status (or create as draft and set status via PUT)
2. Execute the transition via `PUT /api/backend/proposals/:id` with the target status
3. Assert the API response is successful (200)
4. Navigate to the proposals list page and verify the UI reflects the new status (per Pitfall 4 from RESEARCH.md: after API call, reload/navigate before asserting)
