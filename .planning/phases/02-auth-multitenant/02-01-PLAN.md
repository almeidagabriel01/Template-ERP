# Plan 1: Auth Flow E2E Tests — Login, Session, Logout, Custom Claims

## Goal
Playwright tests prove that authentication works end-to-end: a user can log in with email and password, the session persists via the `__session` cookie, logout clears the session, and Firebase custom claims (`tenantId`, `role`, `masterId`) are correct after login.

## Context
Phase 1 established the full test infrastructure: Playwright + Firebase Emulators + seed data + Page Object Model + auth fixtures. Phase 2 builds the first real test suite on top of this foundation. The smoke tests in `e2e/smoke.spec.ts` already prove the login page loads and `authenticatedPage` lands on an authenticated route — this plan adds explicit assertions about session cookies, logout, and custom claims.

Custom claims cannot be read from the browser-side Firebase SDK without internal hackery. Instead, a Node.js helper calls the Auth emulator REST API directly to sign in and decode the JWT payload — a clean, stable approach that doesn't touch the browser.

The `interceptFirebaseRequests()` function in `auth.fixture.ts` already handles emulator URL rewriting for browser-side Firebase calls. No additional interception setup is needed for the new tests.

<read_first>

- e2e/fixtures/auth.fixture.ts -- understand interceptFirebaseRequests (Node-side vs browser-side distinction)
</read_first>
<steps>
1. Create `e2e/helpers/firebase-auth-api.ts`.

2. Define and export `interface SignInResponse`:
   ```typescript
   export interface SignInResponse {
     idToken: string;
     refreshToken: string;
     localId: string;
     email: string;
   }
   ```

3. Define and export `interface TokenClaims`:
   ```typescript
   export interface TokenClaims {
     uid: string;
     email: string;
     tenantId?: string;
     role?: string;
     masterId?: string;
     isSuperAdmin?: boolean;
     [key: string]: unknown;
   }
   ```

4. Export `async function signInWithEmailPassword(email: string, password: string): Promise<SignInResponse>`:
   - POST to `http://127.0.0.1:9099/identitytoolkit.googleapis.com/v1/accounts:signInWithPassword?key=demo-key`
   - Body: `{ email, password, returnSecureToken: true }`
   - Headers: `{ 'Content-Type': 'application/json' }`
   - Throw if response is not ok, including the error message from the body
   - Return the parsed response as `SignInResponse`

5. Export `function decodeJwtPayload(idToken: string): Record<string, unknown>`:
   - Split the token on `.` to get 3 parts: header, payload, signature
   - Base64url decode the payload: replace `-` with `+`, `|` with `/`, then `Buffer.from(payload, 'base64').toString('utf-8')`
   - Parse and return as JSON

6. Export `async function getIdTokenClaims(email: string, password: string): Promise<TokenClaims>`:
   - Call `signInWithEmailPassword(email, password)`
   - Call `decodeJwtPayload(response.idToken)`
   - Return `{ uid: decoded.user_id as string, email: decoded.email as string, tenantId: decoded.tenantId as string | undefined, role: decoded.role as string | undefined, masterId: decoded.masterId as string | undefined, isSuperAdmin: decoded.isSuperAdmin as boolean | undefined }`
   - Keep the spread `...decoded` at the end so raw claims are also accessible

Note: Firebase ID tokens use `user_id` (not `sub`) for the UID in the payload.
</steps>

   - Verify login form is visible again: `await expect(page.locator('input[type="email"]')).toBeVisible()`
