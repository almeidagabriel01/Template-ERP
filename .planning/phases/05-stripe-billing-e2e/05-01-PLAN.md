- `test.afterEach`: Call `restoreTenantState(db, "tenant-beta")` to clean up.
- Test "tenant on free plan at limit is blocked, then upgrade to pro unblocks":
  1. Get idToken via `signInWithEmailPassword`
  2. POST to `${FUNCTIONS_BASE}/v1/proposals` with `Authorization: Bearer ${idToken}` header and minimal valid proposal payload: `{ clientId: "test-client", items: [], niche: "cortinas", status: "draft" }`. Assert response status is **402** (not 403 -- per research Finding 1). Assert response body contains `"code": "PLAN_LIMIT_PROPOSALS_MONTHLY"`.

  4. POST same proposal request again. Assert response status is **201** (unblocked).
  5. If 201 succeeded, delete the created proposal via DELETE `${FUNCTIONS_BASE}/v1/proposals/{id}` to clean up.

**Describe block 2: "BILL-02: subscription.created state allows API call" (per D-03, D-04)**

  1. Get idToken

  4. POST proposal again -- assert 201 (unblocked after simulated subscription.created write)
  5. Clean up created proposal via DELETE

  1. Get idToken

CRITICAL CACHE NOTE: `tenant-plan-policy.ts` has a 30-second in-memory cache per tenantId. Since the emulator keeps one instance alive, sequential tests on the same tenantId may hit stale cache. To handle this:
- Use `test.describe.serial()` to run tests sequentially within each describe block
- Add a `await new Promise(r => setTimeout(r, 31000))` wait between plan state changes if the first attempt gets a stale cached response. BUT FIRST try without the wait -- the plan cache TTL may have already expired between test setup and assertion. Only add the wait if tests fail due to stale cache.
- Alternative: If cache is problematic, fall back to using a unique throwaway tenantId per test (create a fresh tenant doc with a timestamp-based ID). This avoids cache entirely. Implement this fallback if the 30s wait approach makes tests too slow.

The proposal POST payload must satisfy validation. Use a minimal valid payload -- read `proposals.controller.ts` create handler to determine required fields. At minimum it needs `clientId`, `items` (can be empty array `[]`), and niche-specific fields.
