# Plan 2: Route Guards + Tenant Isolation E2E Tests

## Goal
Playwright tests prove that unauthenticated users are redirected to login (route guards), and that Tenant A cannot read or modify Tenant B's data at both the Firestore rules and backend API layers (tenant isolation).

## Context
This plan depends on Plan 02-01 for the `firebase-auth-api.ts` helper (needed to obtain ID tokens for direct API calls in the isolation tests). The route guard tests are the simplest in the phase — they just navigate without auth and verify the redirect. The tenant isolation tests are the most security-critical: they verify the DENY-by-default guarantee that underlies every multi-tenant operation in the system.

The isolation tests hit two enforcement layers:
1. **Firestore security rules** (tested via the Firestore emulator REST API with a signed ID token)
2. **Backend API enforcement** (tested via the Next.js proxy with a signed ID token, targeting a beta document by ID)

Both layers must hold independently — Firestore rules are the last line of defense if the backend has a bug.

<steps>
1. Create `e2e/auth/tenant-isolation.spec.ts`.

2. Import:
   ```typescript
   import { test, expect } from "../fixtures/base.fixture";
   import { signInWithEmailPassword } from "../helpers/firebase-auth-api";
   import { PROPOSAL_BETA_DRAFT } from "../seed/data/proposals";
   import { USER_ADMIN_ALPHA, USER_ADMIN_BETA } from "../seed/data/users";
   ```

3. Write `describe('AUTH-06: Tenant isolation', ...)`.

   **Test: "Firestore denies cross-tenant read via client SDK token"**

   This test calls the Firestore emulator REST API directly from Node.js (not via browser) using alpha's ID token. The Firestore security rules must reject the request because alpha's `tenantId` claim ('tenant-alpha') does not match the document's `tenantId` field ('tenant-beta').

   ```typescript
   test('alpha token cannot read beta proposal from Firestore', async () => {
     // Get alpha's raw ID token
     const { idToken } = await signInWithEmailPassword(
       USER_ADMIN_ALPHA.email,
       USER_ADMIN_ALPHA.password,
     );

     // Attempt to read a beta document directly from Firestore emulator
     const docPath = `proposals/${PROPOSAL_BETA_DRAFT.id}`;
     const firestoreUrl = `http://127.0.0.1:8080/v1/projects/demo-proops-test/databases/(default)/documents/${docPath}`;

     const response = await fetch(firestoreUrl, {
       method: 'GET',
       headers: { Authorization: `Bearer ${idToken}` },
     });

     // Firestore security rules must deny cross-tenant access
     expect(response.status).toBe(403);
   });
   ```

   **Test: "Firestore denies cross-tenant write via client SDK token"**

   Same pattern but with PATCH/update:
   ```typescript
   test('alpha token cannot write to beta proposal in Firestore', async () => {
     const { idToken } = await signInWithEmailPassword(
       USER_ADMIN_ALPHA.email,
       USER_ADMIN_ALPHA.password,
     );

     const docPath = `proposals/${PROPOSAL_BETA_DRAFT.id}`;
     const firestoreUrl = `http://127.0.0.1:8080/v1/projects/demo-proops-test/databases/(default)/documents/${docPath}`;

     const response = await fetch(firestoreUrl, {
       method: 'PATCH',
       headers: {
         Authorization: `Bearer ${idToken}`,
         'Content-Type': 'application/json',
       },
       body: JSON.stringify({
         fields: { title: { stringValue: 'Hacked by Alpha' } },
       }),
     });

     // Firestore security rules must deny cross-tenant write
     expect(response.status).toBe(403);
   });
   ```

   **Test: "backend API returns 4xx when alpha token targets beta proposal"**

   This calls the Next.js backend proxy to test server-side enforcement. The backend uses `req.user.tenantId` (from the token) to filter queries — it will not find the beta proposal because the token says `tenant-alpha`.

   ```typescript
   test('backend API blocks cross-tenant proposal update', async ({ request }) => {
     // Get alpha's ID token
     const { idToken } = await signInWithEmailPassword(
       USER_ADMIN_ALPHA.email,
       USER_ADMIN_ALPHA.password,
     );

     // Try to PUT a beta-owned proposal using alpha's token
     // baseURL is http://localhost:3001 (or 3000 per playwright.config.ts)
     const response = await request.put(
       `/api/backend/proposals/${PROPOSAL_BETA_DRAFT.id}`,
       {
         headers: {
           Authorization: `Bearer ${idToken}`,
           'Content-Type': 'application/json',
         },
         data: { title: 'Cross-tenant attack attempt' },
       },
     );

     // Backend must reject: either 403 (explicit tenant check) or 404 (document not found for this tenant)
     expect([403, 404]).toContain(response.status());
   });
   ```

   **Test: "beta token cannot read alpha proposal from Firestore"** (symmetry check)

   Mirror of the first Firestore test but with beta trying to read alpha's document:
   ```typescript
   test('beta token cannot read alpha proposal from Firestore', async () => {
     const { idToken } = await signInWithEmailPassword(
       USER_ADMIN_BETA.email,
       USER_ADMIN_BETA.password,
     );

     const docPath = `proposals/proposal-alpha-draft`;
     const firestoreUrl = `http://127.0.0.1:8080/v1/projects/demo-proops-test/databases/(default)/documents/${docPath}`;

     const response = await fetch(firestoreUrl, {
       method: 'GET',
       headers: { Authorization: `Bearer ${idToken}` },
     });

     expect(response.status).toBe(403);
   });
   ```

4. Important implementation notes:
   - The Firestore REST API tests are Node.js `fetch()` calls — no browser is involved. `test()` with no `{ page }` destructuring is correct.
   - The backend API test uses Playwright's `request` fixture which handles the baseURL automatically.
   - For the `request` fixture test, Playwright's `request` uses `playwright.config.ts` `baseURL` — confirm it points to `http://localhost:3001` (or the configured port).
   - The `fetch()` calls in isolation tests go directly to emulators — they do NOT go through `base.fixture.ts`'s `setupEmulatorRoutes()` (which only intercepts browser `page` requests). This is intentional: these are Node.js-side calls.
