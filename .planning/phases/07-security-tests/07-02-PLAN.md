Add a new `firestore-rules` job to `.github/workflows/ci.yml`. Insert it after the `performance` job and before the `security` job (or after `security` — order doesn't matter since all 4 jobs run in parallel).

The `firestore-rules` job is lighter than `e2e` — it only needs: Firebase CLI + Firestore emulator (no functions, no Next.js). Use `firebase emulators:exec` to start/stop the Firestore emulator automatically around the Jest run.

**New job to add:**

```yaml
  firestore-rules:
    name: Firestore Rules Tests
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Install Firebase CLI
        run: npm install -g firebase-tools

      - name: Run Firestore rules tests
        run: firebase emulators:exec --only firestore --project demo-proops-test "npm run test:rules"
        env:
          CI: true
```

**Key design decisions reflected in this job:**
- `--only firestore` — only starts the Firestore emulator (port 8080); no Auth/Functions/Storage emulators needed
- `--project demo-proops-test` — matches the projectId used in `initializeTestEnvironment` in the test file
- `firebase emulators:exec` automatically starts the emulator, runs the command, then stops it
- No `functions build` step needed — rules tests don't hit the Functions emulator
- `timeout-minutes: 10` — lighter than e2e (15 min); rules tests are fast
- No artifact upload needed — Jest output to stdout is sufficient; failures fail the job

**Do NOT add a `needs:` dependency** — this job runs parallel to `e2e`, `performance`, and `security`. All 4 jobs block PR merge independently.

**Important:** Confirm the YAML indentation matches the existing jobs in the file (2-space or 4-space — match whatever is already used). Read the full file first to confirm.
